# セッションごとに git worktree / branch を分ける

## ルール

複数の Claude Code セッション（別ウィンドウ・別タブ・並行して起動した会話）が
同時にこのリポジトリで作業する場合、**同じチェックアウト
（`/home/cuzic/rust-nicola` の作業ツリーそのもの）を共有してはならない**。
セッション（あるいは着手するタスク）ごとに、

1. `git worktree add <path> -b <branch-name>` で専用の worktree と
   専用ブランチを作成する。
2. そのセッションでの変更・コミットは専用 worktree の中だけで行う。
3. 作業が一段落したら、通常のブランチと同じように PR 化 / main へのマージを
   検討する。使い終わった worktree は `git worktree remove <path>` で片付ける。

`/home/cuzic/rust-nicola` 直下（メインの作業ツリー）は、ユーザーが対話的に
直接触っている作業や、明示的に「ここで」と指示された単発作業にのみ使う。

## なぜこのルールが必要か（背景）

2026-07-09、`/home/cuzic/rust-nicola` の同一ワークツリー上で少なくとも2つの
Claude Code セッションが並行して動作し、互いに気づかないまま別々のファイルを
編集・コミットしていた（BUG-21 修正セッションと BUG-22 修正セッション）。
一方のセッションが `git add`（明示ファイル指定）しようとした際に、
`git status` で **自分が変更していないファイルの差分**（他セッションの
作業中の編集）が見えてしまい、誤って巻き込んでコミットしてしまう寸前だった。

単一の作業ツリーを共有すると、以下が構造的に起こり得る:

- 片方のセッションの未コミット編集を、もう片方が `git add -A` 等で誤って
  自分のコミットに巻き込む。
- 片方が `git checkout` / `git stash` / `git reset` 等を実行すると、
  もう片方が作業中のファイルが強制的に書き換わる。
- どちらのセッションの変更か区別がつかず、`git blame` や commit 履歴が
  意図を追えなくなる。

`git worktree` を使えば、同じリポジトリ（`.git`）を共有しつつ作業ディレクトリ
そのものは分離できるため、上記の事故が構造的に起きなくなる。ブランチを分ける
ことで、あるセッションの作業が別セッションの意図しないマージ対象に混ざる
こともなくなる。

## 適用範囲

- ユーザーが明示的に「ここで作業して」「このディレクトリで」と指定した場合は
  メイン作業ツリーで進めてよい。
- 自律的に複数タスクを並行させる場合（`Agent` ツールの `isolation: "worktree"`
  を含む）は、このルールに従い必ず worktree を分ける。
- 単一セッション内で複数の独立したタスクを順番に行う分には、このルールは
  適用されない（同一セッション内なので競合しない）。
