# ADR-040: 大規模リファクタリングの段階的遷移戦略

## ステータス

採用済み（ADR-032 Phase 1〜3 実施、2026-05-24〜05-29）

## コンテキスト

ADR-032 で定義した「IME 状態 reducer 4 階層モデル」への移行は、既存の「5 優先度 + sideband guard 7 個」構造を全面的に置き換えるものだった。一括置換を試みると：
- 既存テストが全て壊れる期間が発生する
- 回帰の原因特定が困難になる
- レビュー対象が巨大になり、見落としが増える

## 決定

大規模なアーキテクチャ変更を**フェーズ分割**し、各フェーズ終了時点でテストスイートが通ることを必須条件とする。

ADR-032 の場合の具体的な分割：

**Phase 1（並走期）:** 旧コード（`belief.ime_on` 優先度モデル）を維持しながら、新コード（`shadow_model` reducer）を「差分ログ出力のみ」で並走させる。実際の制御は旧コードが行う。

**Phase 2（補完期）:** 新コードの欠落パスを補完する。`ImeApplySucceeded` / `ImeApplyFailed` の event dispatch を追加し、新旧の状態が一致することを diff ログで確認。

**Phase 3（昇格期）:** 旧コードの出力を新コードの出力で置き換える。`effective_open()` を旧 `belief.ime_on` の読み取り箇所に差し込む。サブフェーズ (3a〜3e) でファイルを分けて段階的に切り替え。

**Phase 3 完了後:** 旧コード（`belief.ime_on` 書き込み箇所、sideband guard）を削除。

## 並走テクニックの要点

- 並走期は「旧コードが正」「新コードは shadow」として、diff が出たらログに記録
- diff ログを CI に取り込み、差異がゼロになったフェーズで昇格判定
- 各サブフェーズを 1 コミットに収め、git bisect で回帰箇所を特定可能に保つ

## 汎用化

このパターンは「状態管理のリファクタリング」全般に適用可能：

```
1. 新旧並走（diff ログ） → 2. 新コード補完 → 3. 段階的昇格 → 4. 旧コード削除
```

特に「グローバルな状態遷移モデルの変更」（Flux, Redux, Actor 等の移行）で有効。

## 結果

- ADR-032 の Phase 1〜3 全て、既存テストを壊さずに完了
- Phase 3e（observer 廃止化）まで進めた後、旧 boolean guard 7 個が全て削除できた
- 段階的遷移により、回帰が発生した際の git bisect が O(log n) コミットで機能した
